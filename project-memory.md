## 1. Project overview

ReliCheck is a web-based survey credibility platform: surveys, classroom tests, and uploaded response datasets feed into a deep analytics stack (reliability, validity, factor analysis, IRT, regression, mediation, moderation, multilevel modeling, equity analysis, trends), and every dashboard is paired with an AI narrator that translates the math into plain language. The product positioning shifted in Phase 49 from "survey platform with stats" to "survey credibility platform" with the builder framed as one of several inputs alongside upload-survey and upload-data flows.

The source tree lives locally at `/Users/deastonbrooks/Dropbox/Claude Work/Projects/relicheck` (mounted via `mcp__cowork__request_cowork_directory(path='/Users/deastonbrooks/Dropbox/Claude Work')` at session start; don't pop the folder picker). The Cowork workspace root is `/Users/deastonbrooks/Dropbox/Claude Work`. Bash operations run under `/sessions/<session-id>/mnt/Claude Work/Projects/relicheck`.

Tech stack is intentionally simple. Frontend is hand-rolled HTML, CSS, and JS with no build step, so what is in source is what ships. The backend is plain PHP with a MySQL database on IONOS shared hosting, accessed through phpMyAdmin. Authentication is email/password plus Google Sign-In; Google Sheets and Drive integration shipped as Phase 12. There is no composer/vendor directory on disk, no Node/npm pipeline, and no framework.

There are two parallel app shells. The classic shell, `app.html`, is the single canonical file Donald edits and uploads. By 2026-05-17 it had grown past 20,000 lines and houses the full analytics stack, Builder, Distribute view, Account modal, and a 12-tab analytics surface. The classic shell is still the production app; app-2026 has reached feature parity in most areas but ships as a parallel surface so users can opt in.

The 2026 redesign moved roughly 30 pages onto `styles-2026.css` (committed 2026-05-08); the rest sit on the legacy `styles.css`. ## 2. User profile and working preferences

Donald Easton-Brooks deploys this entire system by hand, runs his own schema migrations through phpMyAdmin, and uploads through FileZilla. He has a clear voice and a precise set of operational preferences. Treat the rules below as load-bearing.

## 3. Deployment and infrastructure

ReliCheck runs on IONOS shared hosting with several quirks worth recording explicitly so they don't bite again.

**Cron is on cron-job.org**, not IONOS. The shared plan doesn't expose a real Linux crontab. Active scheduled URLs include `/api/email/queue_run.php` (every minute), `/api/admin/cron/survey_activity.php` (daily), `/api/admin/cron/recompute_health.php` (daily), `/api/admin/cron/send_reminders.php` (daily), `/api/cron-fire-pulse.php` (hourly), and `/api/cron-fire-calendar-followups.php` (hourly). All use the same `email_cron_key` value from `api/_config.php`. The key value lives only on the server; never ask Donald to paste it back in chat.

**PHP and MySQL run in different timezones on this host.** PHP's `date.timezone` is in EDT while MySQL's `NOW()` returns UTC (~4-hour drift). Never write a PHP-computed datetime to a column compared to `NOW()`; use `DATE_ADD(NOW(), INTERVAL N MINUTE)` inline in the INSERT/UPDATE. Existing patterns work only because they explicitly parse with `' UTC'` suffix (e.g., `password_resets`, `admin_password_resets`); convert to `DATE_ADD` next pass.

**SHOW LIKE with a placeholder fails on IONOS MySQL.** `$pdo->prepare('SHOW TABLES LIKE :n')` either throws an uncaught `PDOException` or silently returns false. Use `$pdo->query("SHOW TABLES LIKE 'literal_string'")` with the value sanitized inline.

**AddType `application/x-httpd-php .html` does NOT work in `.htaccess` on IONOS.** Phase 149 tried this; the directive strips the default text/html MIME type and the server starts serving .html files as `application/octet-stream`. Emergency rollback inlined the nav back into every page. The Phase 150 solution uses JavaScript injection via `nav.js` and `footer.js`. Test any future server-config change on ONE page first before bulk-converting.

**FileZilla cross-contamination is a real risk.** When an endpoint silently misbehaves while auth/config look healthy, suspect a wrong-file upload to that path. The 2026-05-08 promo-codes outage came from `api/promo/list.php` having been overwritten with the contents of `api/responses/list.php`. Fetch the raw response body and check whether it matches an error string from a different endpoint; if so, re-upload from the build.

**PHP closing tag `?>` terminates `//` comments AND exits PHP mode.** A stray `?>` inside a docblock annotation like `<string?>` collapses the file to an inline node from that line to EOF; the server returns HTTP 200 with the PHP source as the response body. When an endpoint returns HTTP 200 with PHP source as the body, grep the file for `?>` FIRST before chasing config, encoding, or upload mode.

**ASCII-only in PHP source.** No em dashes, en dashes, ellipses, `>=`/`<=` glyphs, or curly quotes anywhere in PHP files. Non-ASCII bytes in PHP source on IONOS is a suspected cause of execution failure independent of the closing-tag bug.

**Deployment manifest.** Top-level `.html`, `styles.css`, `styles-2026.css`, `favicon.svg`, `.htaccess`, `404.html`, image assets, and the entire `samples/`, `posts/`, `case-studies/`, `help/`, `dashboards/`, and `api/` folders ship via FileZilla. Do NOT upload `.DS_Store`, `SETUP.md`, `db/` (phpMyAdmin only), `test_data/`, raw PNGs alongside optimized JPGs, `api/_config.example.php`, or anything under `/preview/`. Use FileZilla SFTP with comparison mode set to "File size and modification time"; overwrite with "newer source replaces target"; hard-refresh after upload. Schema migration files at `db/schema_phaseN.sql` run through phpMyAdmin in phase order, and every migration handoff must lead with "click `dbs15641829` in the phpMyAdmin sidebar first" because Donald has been bitten by the `#1046 No database selected` error.

## 4. Data model and code conventions

A handful of conventions are load-bearing across the codebase.

**`q.options` is a flat string array, not objects.** For single-choice and multi-select questions, `q.options` is an array of strings where the numeric index is the stored response value and the string is the display label. Analytics functions that build `optMap` must type-check; expecting `.value`/`.label` produces "Group 0/1/2" labels instead of names. Fixed in `_compareDetectGroupCandidates` (Compare and Equity Gaps) and `_mlmGroupCandidates` (MLM) on 2026-05-15. Any new categorical-group analytics must apply the same pattern.

**Dataset vs survey state distinction.** When the user opens a dataset for analytics via `openDataset`, state is set to `state.datasetId=<id>` with `state.surveyId=null` and `state.survey` built from `datasetToSurveyShape(ds)`. Dataset rows live in `datasets.data` JSON, not in the `responses` table. AI endpoints keyed on `survey_id` cannot serve dataset rows directly. For new AI features that need to work in both modes, the established pattern is a parallel stateless endpoint that accepts raw inputs (precedent: `/api/ai/extract-themes-from-texts.php` parallels the survey-side `extract-themes.php`). The client gathers the inputs from `state.responses` in dataset mode while survey mode hits the survey_id-based endpoint.

**Hybrid placement pattern.** When a feature serves more than one user moment (pre-publish AND post-data, pre-distribution AND ongoing review), don't collapse it to a single home. Phase 144b established the pattern: primary card at the moment-of-decision view (e.g., Survey Readiness on Distribute), full deep view kept where the analysis chrome already lives (Analyze), and a ctx-bar pill as the cross-view discoverability anchor. All three surfaces call the same `build*Snapshot()` so the numbers never drift.

**Tab classification rule.** Surfaced top-row analytics tabs are survey-quality reads (about the survey itself or the data-collection process: Readiness, Reliability, Validity, Open-Ended, Quality, Completion, Strength Index). The More analyses dropdown holds data-outcome reads (Compare, Pre/Post, Subgroups, Predictors, Key Drivers, IRT, MLM, Trends). Phase 142 originally surfaced Trends and violated the rule; Phase 144 moved it to the dropdown.

**Analytics dashboard consistency rule.** Every analytics tab shares the same chrome: card-wrapped header with H2 (Fraunces, 24px), narrator placement in the main content column anchored just above the primary analysis card via `insertTabNarratorCard` with a `before:` reference, Ask-data chat card in the right rail, no legacy "Generate summary" rail card, KPI tiles/data tables/scroll-x wrappers/fit-indicator colors all using existing classes (`.fa-kpi-tile`, `.fa-kpi-label`, `.fa-kpi-num`, `.fa-kpi-sub`, `.data-table`, `.fa-section`, `.scroll-x`). Setup cards (Compare/Pre-Post/Subgroups/Invariance) appear AFTER the narrator. The fastest sanity check when building a new tab is to switch between it and the Subgroups tab and confirm the header row, narrator card, and rail look identical.

**Narrator consolidation patterns.** All narrators share a single contract `{ tone, tone_label, headline, paragraph, highlights[] }` (Phase 54). Phase 93 added optional `affected_items[]` to enable a "Show affected items" button that scrolls to and flashes the matching rows; Phase 98 added optional structured `findings[]` and `recommendations[]` for richer narrator outputs (currently used only by the Test analytics narrator, opt-in per-tab). Cache keys live in localStorage under `relicheck.narrator.<tab>.v1.<survey_id>.<response_count>...` so refetches happen automatically when the underlying data changes. Phase 92 added a Copy summary button to every narrator card. Phase 105 added the surfaced-tab hero pattern that wraps the narrator card with the SSI ring teaser on the right plus a Recommended Next Step card at the bottom. Phase 109 generalized the sub-tab pattern across every analytics tab via `_renderSubTabBar(root, key, items)` and `_applySubTabVisibility(root, key)`.
